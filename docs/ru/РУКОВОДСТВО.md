# Документация: торговая платформа (MVP backend)

Монолитный backend на **Python 3.12** и **FastAPI**: пользователи, JWT, API-ключи бирж, боты сетки на **Binance Futures** (через **ccxt**), фоновые задачи (**Celery**), кэш и pub/sub (**Redis**), БД (**PostgreSQL**), уведомления в **Telegram** (aiogram).

Интерактивная спецификация API после запуска: **Swagger UI** — `http://localhost:8000/docs`, **ReDoc** — `http://localhost:8000/redoc`.

## Быстрый старт

1. Установите **Python 3.12+**.
2. Создайте виртуальное окружение и зависимости:

```bash
pip install -e ".[dev]"
```

3. Сгенерируйте ключ шифрования **Fernet** для секретов API-ключей:

```bash
python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

Полученную строку укажите в `ENCRYPTION_KEY` в файле `.env`.

4. Скопируйте `.env.example` в `.env` и заполните переменные (минимум: `DATABASE_URL`, `REDIS_URL`, `JWT_SECRET`, `ENCRYPTION_KEY`).

5. Поднимите **PostgreSQL** и **Redis** (или используйте Docker Compose — см. ниже).

6. Примените миграции:

```bash
alembic upgrade head
```

7. Запустите API:

```bash
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

Проверка: `GET http://localhost:8000/health` → `{"status":"ok"}`.

## Переменные окружения

| Переменная | Назначение |
|------------|------------|
| `DATABASE_URL` | Строка SQLAlchemy async, например `postgresql+asyncpg://user:password@localhost:5432/app_db` |
| `REDIS_URL` | Redis для кэша, pub/sub WebSocket, лимитов вызовов к бирже |
| `CELERY_BROKER_URL` / `CELERY_RESULT_BACKEND` | Очередь Celery (часто отдельные DB в Redis) |
| `JWT_SECRET` | Секрет подписи access/refresh JWT |
| `ENCRYPTION_KEY` | Ключ Fernet для шифрования `api_key` / `api_secret` в БД |
| `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` / `GOOGLE_REDIRECT_URI` | OAuth Google (опционально) |
| `TELEGRAM_BOT_TOKEN` | Токен бота для уведомлений и привязки по коду |
| `BINANCE_TESTNET` | `true` / `false` — тестовая сеть Binance Futures |
| `SMTP_*` | Отправка писем сброса пароля (опционально) |
| `MAX_API_KEYS_*`, `MAX_ACTIVE_BOTS_*` | Лимиты по ролям пользователя (см. `app/core/config.py`) |

Полный список с дефолтами — в [`app/core/config.py`](../../app/core/config.py).

## Docker Compose

Из корня проекта:

```bash
docker compose up --build
```

Сервисы:

| Сервис | Описание |
|--------|----------|
| `db` | PostgreSQL, хост в сети compose: `db`, порт 5432 |
| `redis` | Redis |
| `backend` | API: миграции Alembic при старте, затем Uvicorn |
| `worker` | Celery worker |
| `bot-engine` | Долгоживущий процесс супервизора ботов |
| `telegram` | Опционально: `docker compose --profile telegram up` (нужен `TELEGRAM_BOT_TOKEN`) |

Перед запуском задайте **`ENCRYPTION_KEY`** в окружении (например через `.env` или переменные CI).

Строка БД в compose уже настроена под сервис `db` (см. `docker-compose.yml`).

## Основные HTTP-маршруты

Базовый URL в примерах: `http://localhost:8000`.

### Аутентификация (`/auth`)

| Метод | Путь | Описание |
|-------|------|----------|
| POST | `/auth/register` | Регистрация (email, пароль) |
| POST | `/auth/login` | Вход |
| POST | `/auth/refresh` | Обновление пары токенов по refresh |
| GET | `/auth/google` | URL авторизации Google (если OAuth настроен) |
| POST | `/auth/google` | Обмен `code` на токены |
| POST | `/auth/telegram` | Вход через Telegram Login Widget (проверка `hash`) |
| POST | `/auth/reset-password` | Запрос письма со сбросом пароля |
| POST | `/auth/reset-password/confirm` | Новый пароль по одноразовому токену |

В защищённых запросах передавайте заголовок:  
`Authorization: Bearer <access_token>`.

### Пользователь (`/user`)

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/user/me` | Текущий пользователь |
| PATCH | `/user/update` | Имя, флаги уведомлений Telegram |
| POST | `/user/telegram/link-code` | Выдача **6-значного кода** для привязки Telegram (вводится в боте) |

### API-ключи (`/api-keys`)

| Метод | Путь | Описание |
|-------|------|----------|
| POST | `/api-keys` | Добавить ключ (секреты шифруются в БД) |
| GET | `/api-keys` | Список (секреты не отдаются, ключ маскируется) |
| DELETE | `/api-keys/{id}` | Мягкое удаление |

Лимит количества ключей зависит от **роли** пользователя.

### Боты (`/bots`)

| Метод | Путь | Описание |
|-------|------|----------|
| POST | `/bots` | Создать бота (MVP: только `GRID_FUTURES`) |
| PATCH | `/bots/{id}` | Обновить конфиг сетки (`config`), `config_version` увеличивается на сервере |
| GET | `/bots/active` | Активные боты |
| GET | `/bots/stopped` | Остановленные |
| GET | `/bots/history` | События (опционально `?bot_id=`) |
| POST | `/bots/{id}/stop` | Стоп: отмена ордеров, статус остановлен |
| POST | `/bots/{id}/close` | Закрытие: отмена ордеров + закрытие позиции |

### Администрирование (`/admin`)

Требуется JWT пользователя с ролью **SUPPORT**, **ADMIN** или **SUPERADMIN** (см. `UserRole`).

| Метод | Путь | Кто | Описание |
|-------|------|-----|----------|
| GET | `/admin/users` | SUPPORT+ | Список пользователей (`skip` / `limit`) |
| PATCH | `/admin/users/{id}/role` | ADMIN, SUPERADMIN | Смена роли пользователя |

### Рынок (`/market`)

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/market/symbols` | Символы биржи, отфильтрованные по ботам пользователя (не `CLOSED`) |

### Служебное

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/health` | Проверка живости сервиса |

## WebSocket

Подключение: **`WS /ws?token=<access_jwt>`** (query-параметр — тот же access token, что и в `Authorization`).

События публикуются в Redis с паттерном каналов `user:{user_id}:*` (например, `user:1:bots`). Клиент подписывается через `PSUBSCRIBE` на стороне сервера и получает JSON-тексты в WebSocket.

## Бот-движок и сетка

- Процесс **`bot-engine`** периодически подгружает активные боты и для каждого запускает цикл: синхронизация ордеров/позиции с биржей, при необходимости выставление сетки с **детерминированными `client_order_id`** (идемпотентность).
- Автоматический **перезапуск сетки** после полного «обнуления» позиции и ордеров включается только флагом **`auto_restart: true`** в JSON-конфиге бота (по умолчанию `false`, чтобы избежать нежелательных циклов на пустом рынке).
- Конфиг сетки (`GRID_FUTURES`): символ, направление LONG/SHORT, начальный объём, число уровней, шаг в процентах, режим объёма (linear / exponential / fixed), опционально `start_price`.

Подробности реализации — в [`app/bot_engine/loop.py`](../../app/bot_engine/loop.py) и [`app/strategies/grid.py`](../../app/strategies/grid.py).

## Безопасность

- Пароли хранятся как хеш (bcrypt через passlib).
- Секреты API-ключей в БД — **только в зашифрованном виде** (Fernet), ключ — `ENCRYPTION_KEY`.
- JWT: короткоживущий access и refresh; refresh привязан к записи в таблице refresh-токенов.
- HTTP rate limit (slowapi) на отдельных эндпоинтах; отдельно — лимит исходящих запросов к бирже на **API-ключ** через Redis.

### Частично или упрощённо относительно плана

| Тема плана | Как сейчас в коде |
|------------|-------------------|
| **FSM** | При авто-рестарте сетки (`auto_restart`) выставляется **`RESTARTING`**, затем снова **`RUNNING`**. Отдельных модулей `fsm.py` / `recovery.py` нет. |
| **Мониторинг Binance** | User Data Stream WebSocket **не** используется — только **polling** ордеров/позиций через ccxt. |
| **Fault tolerance** | Цикл бота: backoff при ошибке + `ERROR` + retry ccxt на сетевых сбоях биржи. Reconnect **клиентского** WebSocket Redis и Binance user stream **не** реализованы. |
| **Идемпотентность** | В цикле сетки: строка в БД без `exchange_order_id` → повторная отправка на биржу; уникальный `client_order_id`. Закрытие позиции при **close** — market с **`close_position_order_id`**. Проверка внутри `ExchangeAdapter.place_order` по БД **не** делается (на уровне движка/сервиса). |
| **Частичные исполнения** | Обновление по **открытым** ордерам; user data stream **нет**. |
| **Audit log** | Не покрыты все сценарии (например системный вход админа без отдельного эндпоинта). Смена роли пишется в audit. |

### Итог

Сценарий **регистрация → ключ → бот → движок → REST/WS/Telegram (с worker)** закрыт для MVP. Остаётся зазор к «идеальному» плану: **Binance user stream**, идемпотентность на уровне адаптера с БД, полный audit и восстановление `recovery.py`.
